1️⃣ The "Sprawl" Problem

Em muitos ambientes corporativos, é comum encontrar uma relação 1:1 entre serviços e API Proxies. Isso leva ao que identificamos como "Proxy Sprawl": um ambiente onde proxies são criados para cada serviço técnico individual, independentemente de seu domínio de negócio.

Sem uma autoridade central de governança ou convenções de nomenclatura padronizadas, esses ambientes rapidamente se tornam não governados e insustentáveis. À medida que o volume de integração cresce, a complexidade de gerenciar, proteger e monitorar centenas de proxies individuais cria um gargalo operacional significativo.

⚠️ Este padrão não é teórico: DCRP foi desenvolvido como uma resposta direta a essas lacunas de governança do mundo real. Após extensa validação arquitetural, esta metodologia foi formalizada para fornecer uma solução repetível para a Comunidade SAP Integration.

2️⃣ What is DCRP (Domain Centric Routing Pattern)?

O Domain-Centric Routing Pattern (DCRP) é um padrão de design arquitetural que consolida múltiplos serviços backend em um único API Proxy orientado ao domínio de negócio. Residindo na Camada de Gateway, DCRP governa como os serviços backend são expostos e roteados sem necessitar mudanças em seus protocolos subjacentes (REST ou SOAP).

Em vez de organizar APIs em silos técnicos, DCRP os alinha com os domínios de negócio da organização. Esta abordagem reduz significativamente a fragmentação de APIs e o overhead de manutenção, permanecendo totalmente compatível com as ofertas de API Product da SAP.

🏛️ Os Três Pilares do DCRP

🎯 Organização Domain-Centric

APIs agrupadas por domínio de negócio (Sales, Finance, Logistics)

NÃO por sistema técnico (Salesforce, SAP, D365)

🗺️ Roteamento Metadata-Driven

KeyValueMap (KVM) armazena mapeamentos path→backend

Mudanças em 30 segundos sem redeployment

🔌 Gateway Protocol-Agnostic

Suporta backends REST e SOAP

Sem lock-in de protocolo

🏗️ DCRP Architecture Overview: The Power of Dynamic Routing

O diagrama abaixo ilustra como o Domain-Centric Routing Pattern (DCRP) transforma o fluxo de integração. Em vez de um panorama fragmentado de proxies individuais, consolidamos a conectividade através de uma camada de gateway unificada e inteligente.

⚙️ O Workflow Técnico:

  1. Ponto de Entrada Unificado: Múltiplos sistemas consumidores—como Salesforce, Dynamics 365 e Vendor CRMs—conectam através de um único Sales Domain APIM Proxy.
  2. O Motor DCRP: Dentro do SAP API Management Gateway, a requisição é processada pelo "Cérebro" do padrão, que executa um KVM Lookup e executa lógica JavaScript para resolver o backend alvo dinamicamente.
  3. Orquestração Segura: A requisição é roteada via túnel seguro para o iFlow apropriado do Cloud Platform Integration (CPI).
  4. Flexibilidade de Backend: CPI lida com a orquestração final com sistemas alvo (e.g., S/4HANA, Legacy Systems, ou SAP ECC) usando protocolos como ODATA, SOAP ou HTTPS.
  5. Governança em Escala: Este modelo elimina a necessidade de proxies separados por integração, permitindo monitoramento centralizado, auditoria e aplicação de segurança através do Analytics Collector e uma camada de Governança global.

🧠 The Technical Foundation: KVM + JavaScript

KeyValueMap (KVM) | A "Memória"

O KVM é uma tabela de lookup simples que armazena suas regras de roteamento.

  • Lógica Desacoplada: Mapeia cpi-packagename, iflowname
  • Atualizações Instantâneas: Mude uma URL backend no KVM, e o proxy atualiza imediatamente—sem deployment necessário
  • Data-Driven: Regras de negócio armazenadas como configuração, mantendo o código limpo

JavaScript Policy | O "Cérebro"

A política JS executa a lógica baseada no que encontra na "Memória".

  • Roteamento Dinâmico: Busca o destino do KVM e constrói a URL final
  • Governança: Injeta automaticamente headers de rastreamento como X-SenderId e X-CorrelationId para auditoria
  • Fail-Safe: Implementa tratamento estruturado de erros, garantindo que se uma rota estiver ausente, o usuário recebe uma mensagem de erro clara em vez de "falha silenciosa"

3️⃣ Implementation: Configuring the DCRP API Proxy

Esta seção define a configuração para o ProxyEndpoint, Routing Rules, e propriedades Target Endpoint necessárias para deployar uma arquitetura domain-centric.

📝 3.1 - The Entry Contract: One API Proxy per Business Domain

DCRP elimina proxy sprawl ao consolidar múltiplas aplicações técnicas em um único contrato de entrada baseado no domínio de negócio. Isso garante que cada área de negócio, como Sales ou Finance, tenha exatamente um API Proxy governado.

🏷️ 3.2 - Naming Convention Strategy

Uma convenção de nomenclatura padronizada fornece contexto arquitetural imediato. DCRP usa a seguinte estrutura:

Padrão: [Provider]-[Pattern]-proxy-domain-[BusinessProcess]-[subprocess]

Exemplo: cpi-dcrp-proxy-domain-sales-o2c

  • cpi: Identifica o provedor (SAP Cloud Integration)
  • dcrp: Define a metodologia de roteamento
  • sales: Identifica o domínio/processo de negócio (O2C)
  • o2c: sap-subprocess

🎯 3.3 - Strategic Benefit: Subprocess Segregation

Quebrar a nomenclatura ao nível de subprocesso (e.g., o2c para Order-to-Cash) é uma estratégia deliberada de segregação ao invés de um simples exercício de rotulagem.

Ao isolar por subprocesso, você limita o escopo de cada API Proxy. Isso previne a "Armadilha do Proxy Monolítico", garantindo que atualizações em um fluxo específico—como Warehouse Management—não arrisquem a estabilidade de todo o domínio Logistics.

⚙️ 3.4 - Endpoint, Routing Configuration, Value Mapping

As Routing Rules atuam como um gatekeeper, garantindo que apenas caminhos autorizados alcancem o motor dinâmico. Neste exemplo, estamos trabalhando apenas com o processo de orders.

Proxy Endpoint Configuration

  • Route Rule Name: orders
  • Condition: proxy.pathsuffix MatchesPath "/orders/**"

💡 Analogia: Imagine sua API como um grande prédio de escritórios. Esta "Condição" é o segurança na recepção checando os rótulos na correspondência recebida.

  • proxy.pathsuffix: O "endereço" escrito no envelope após o nome principal do prédio
  • MatchesPath: O guarda perguntando: "Este endereço parece com o que estou procurando?"
  • "/orders/**": O nome específico do "Departamento". Os asteriscos duplos (**) significam "não me importo com o que mais está escrito depois de 'orders', desde que comece com isso"

🔍 Understanding the iflowname String

A entrada iflowname é a "tabela de roteamento" para o DCRP. Ela contém pares consistindo do Semantic Path Suffix (usado pelo consumidor) e o Technical iFlow Name (o alvo no CPI). Esta string é parseada pelo "Cérebro" JavaScript, então manter o padrão é crítico.

Exemplo de Configuração:

iflowname: inbound:SO.01.Iflow.Sales.Order.Inbound,tracking:SO.02.Iflow.Sales.Order.Tracking

Mapeia o sufixo da URL HTTP pública para o endpoint técnico específico do CPI:

  • inbound → Roteia para /inbound
  • tracking → Roteia para /tracking

⚠️ Nota: O JavaScript suporta quase todos os tipos de delimitadores, mas você não pode usar dois pontos (:) porque isso é a separação do http path URL do SAP CPI e o nome do iFlow. Entre os registros você pode usar qualquer um, minha sugestão é vírgula (,)

🔄 The Full Configuration of API Proxy

Fluxo de Processamento

  1. Condition: O Proxy Endpoint valida o path (e.g., proxy.pathsuffix MatchesPath "/orders/**")
  2. Lookup: O "Cérebro" extrai a string iflowname do KVM
  3. Parsing: O JavaScript divide a string usando o delimitador , para extrair pares individuais
  4. Generation: O motor corresponde o sufixo recebido (e.g., /inbound) e gera a URL CPI Target final
  5. End URL: https://<cpi>/orders/inbound e quaisquer outros valores dos registros KVM

🏗️ ARCHITECT'S DEEP DIVE: WHY SUBDOMAIN SEGREGATION?

⚠️ Contexto: O Desafio de Escalabilidade do KVM

Você pode estar se perguntando:

"Se eu armazenar todos os meus iFlows em um único KVM como uma string concatenada como cpihttpPath:iflowName,, isso não se tornará uma string gigantesca?"

Resposta: Sim, exatamente. E este é o problema central que levou ao pivot arquitetural.

🔍 A Descoberta: Performance de Storage

Scenario Size Entries Storage Runtime Concern
Small Domain ~5 KB 40 entries ✅ Within limits Parsing overhead manageable
Medium Domain ~10 KB 53 entries ✅ At documented limit JavaScript performance degrades
Large Domain ~15-30 KB 100+ entries ⚠️ Exceeds guideline Bottleneck: O(n) parsing, not storage

💡 Descoberta-Chave: Capacidade de armazenamento não é a restrição - performance de parsing JavaScript em escala é.

⚡ 3.6 - The Routing Engine: Execution Flow

Toda requisição passa por três estágios:

Stage 1: Proxy Endpoint Pre-Flow

  • Políticas de segurança (OAuth2, API Key)
  • Hacking Thread JavaScript
  • Validação de requisição (payload size, content type)
  • Controle de tráfego (Spike Arrest, Quotas)
  • Rejeição imediata se políticas falharem

Stage 2: Routing Rules

  • Avalia: proxy.pathsuffix MatchesPath "/orders/**"
  • Match → forward para Target Endpoint
  • No match → erro 404

Stage 3: Target Endpoint Pre-Flow

  • DCRP Brain executa aqui (último ponto para modificar target.url)

🧠 3.7 - The DCRP Brain: Policy Stages

O motor de roteamento executa através de quatro estágios de política no Target Endpoint - PRE-Flow:

  1. Key Value Mapping: Recupera httppath, iflowname, e sapprocess
  2. JavaScript Routing: Executa verificação de variáveis, validação e injeção de headers
  3. RaiseFaultException: Bloqueia o processo em caso de exceções, variáveis ausentes ou configurações incorretas do KVM
  4. Statistics Collection: Captura variáveis de dimensão customizadas para o SAP APIM Dashboard

O padrão DCRP transforma o gateway em um roteador programável de alta escala ao desacoplar configuração de execução e mover lógica de roteamento para metadados.

function resolveDcrpRouting() {
    try {
        var cpiHost = context.getVariable("target.cpi.host");
        var cpiBasepathRaw = context.getVariable("target.cpi.basepath");
        var iflowname = context.getVariable("iflowname");
        var pathSuffix = context.getVariable("proxy.pathsuffix");
        var senderId = context.getVariable("request.header.X-SenderId");
        var correlationId = context.getVariable("request.header.X-CorrelationId");
        var queryParams = context.getVariable("request.querystring") || "";

        if (!cpiHost || !cpiBasepathRaw || !iflowname || !pathSuffix) {
            context.setVariable("routing.failed", true);
            context.setVariable("routing.error", "MISSING_REQUIRED_VARIABLES");
            throw new Error("DCRP: Missing required variables for routing");
        }

        // ========================================
        // SECTION 1.5: EXTRACT PATH SEGMENTS
        // ========================================

        var cleanPathSuffix = pathSuffix.replace(/^\/+/, '');
        var segments = cleanPathSuffix.split('/');
        var firstSegment = segments[0];
        var remainingPath = segments.slice(1).join('/');

        // Build dynamic basepath: /http/{firstSegment}
        var cpiBasepath = "/http/" + firstSegment;

        // ========================================
        // SECTION 2: FLEXIBLE DELIMITER DETECTION
        // ========================================

        var supportedDelimiters = [',', ';', '|', '/', '\\', '[', ']', '(', ')', '-', '_'];
        var delimiter = null;

        for (var i = 0; i < supportedDelimiters.length; i++) {
            if (iflowname.indexOf(supportedDelimiters[i]) !== -1) {
                delimiter = supportedDelimiters[i];
                break;
            }
        }

        if (!delimiter) {
            context.setVariable("routing.failed", true);
            context.setVariable("routing.error", "NO_DELIMITER_FOUND");
            throw new Error("DCRP: No valid delimiter found in KVM value");
        }

        // ========================================
        // SECTION 3: KVM PARSING & PATH MATCHING
        // ========================================

        var entries = iflowname.split(delimiter);
        var routes = [];

        for (var j = 0; j < entries.length; j++) {
            var entry = entries[j].trim();
            if (entry.length === 0) continue;

            var parts = entry.split(':');
            if (parts.length !== 2) {
                context.setVariable("routing.failed", true);
                context.setVariable("routing.error", "INVALID_KVM_FORMAT");
                throw new Error("DCRP: Invalid KVM format - expected 'path:iflowname', got: " + entry);
            }

            routes.push({
                path: parts[0].trim(),
                iflow: parts[1].trim()
            });
        }

        routes.sort(function(a, b) {
            return b.path.length - a.path.length;
        });

        var matchedRoute = null;

        for (var k = 0; k < routes.length; k++) {
            var routePath = routes[k].path.replace(/^\/+/, '');

            if (remainingPath === routePath || remainingPath.indexOf(routePath + '/') === 0) {
                matchedRoute = routes[k];
                break;
            }
        }

        if (!matchedRoute) {
            context.setVariable("routing.failed", true);
            context.setVariable("routing.error", "ROUTE_NOT_FOUND");
            context.setVariable("routing.pathsuffix", pathSuffix);
            throw new Error("DCRP: No matching route found for path: " + pathSuffix);
        }

        // ========================================
        // SECTION 4: GOVERNANCE HEADER INJECTION
        // ========================================

        if (senderId) {
            context.setVariable("request.header.X-SenderId", senderId);
        }

        if (correlationId) {
            context.setVariable("request.header.X-CorrelationId", correlationId);
        }

        // ========================================
        // SECTION 5: DYNAMIC URL CONSTRUCTION
        // ========================================

        var targetUrl = cpiHost + cpiBasepath + "/" + matchedRoute.path;

        if (queryParams && queryParams.length > 0) {
            targetUrl += "?" + queryParams;
        }

        context.setVariable("target.url", targetUrl);
        context.setVariable("target.copy.pathsuffix", false);
        context.setVariable("target.copy.queryparams", false);

        // ========================================
        // SUCCESS: SET ROUTING METADATA
        // ========================================

        context.setVariable("routing.success", true);
        context.setVariable("routing.matched.path", matchedRoute.path);
        context.setVariable("routing.matched.iflow", matchedRoute.iflow);
        context.setVariable("routing.target.url", targetUrl);
        context.setVariable("routing.selected.basepath", cpiBasepath);

        context.setVariable("idiflow", matchedRoute.iflow);
        context.setVariable("idcorrelation", context.getVariable("request.header.X-CorrelationId"));
        context.setVariable("idsender", senderId || "UNKNOWN");

    } catch (error) {
        context.setVariable("routing.failed", true);
        context.setVariable("routing.exception", error.message || error.toString());

        print("DCRP Routing Error: " + error.message);
        print("PathSuffix: " + context.getVariable("proxy.pathsuffix"));
    }
}

// Execute the routing function
resolveDcrpRouting();

💡 TL;DR para Não-Desenvolvedores: Este JavaScript executa 4 tarefas:

  • Extrai segmentos de path (e.g., /inbound → httppath cpi inbound)
  • Busca metadados de roteamento do KVM
  • Constrói URL CPI final dinamicamente
  • Injeta headers de governança (X-SenderId, X-CorrelationId)

🎯 Conclusion

O Domain-Centric Routing Pattern (DCRP) representa uma mudança fundamental na forma como organizações enterprise gerenciam integração em escala. Ao consolidar múltiplos serviços técnicos em proxies únicos orientados ao domínio de negócio, DCRP elimina a proliferação de infraestrutura enquanto habilita governança centralizada.
